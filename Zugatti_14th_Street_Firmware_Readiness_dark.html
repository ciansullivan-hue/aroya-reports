Recommended firmware-push sequence

  1. Restore the backbone & broken devices first. Work the Broken / down list (sinks → gateways → repeaters → nodes → probes). A node that can't be reached can't be flashed, and a probe that's dark because its node dropped will return on its own once the node rejoins.
  2. Fix the "Probably won't" nodes before you push. These are reporting but on a critical battery or radio margin — flashing them as-is risks bricking them mid-write or losing them on the rejoin. Replace batteries / add repeater coverage, confirm they recover, then move them into the batch.
  3. Update the line-powered backbone (repeaters, gateways, sinks). They take firmware reliably on wired power and should be current before the battery fleet so the mesh path is stable for OTA.
  4. Push the "Will sail through" and "Will work" nodes. The bulk of the fleet — update in normal batches and confirm each rejoins.
  5. Return to the "Might work" nodes after their troubleshooting step. Once the noted fix is done and the node holds a stable link, include it.

Methodology. Per-device chart telemetry pulled from the AROYA public API (/devices/<id>/chart/) over a 30-day window, sampled at four 2-day windows (most-recent + three evenly spaced) to respect API rate limits. Gap statistics are computed within each window so the spacing between sampled windows is never mistaken for a dropout. Mesh nodes are graded on battery (Li-SOCl₂ p05), RSSI (p05), link quality, mesh travel time (p95) and in-window dropouts. TEROS-12 substrate probes and Atmos-14 climate sensors carry no radio/battery of their own — they report through their host node, so they are graded on data continuity and the node's health. PoE backbone (repeater / gateway / sink) is line-powered and low brick-risk; its only failure mode here is being offline. Thresholds: battery crit<3.0V / warn<3.2V; RSSI crit<-95 / warn<-85 dBm; link crit<40 / warn<60; travel crit>60 / warn>30 s. "Last seen" > 48 h (outside the most-recent sampled window) is treated as down. Flags use the worst 5th/95th percentile, not the mean, to catch intermittent failures.

Note: This is a triage tool to plan the firmware rollout — a flag means "look before you push", not an automatic replacement order.
